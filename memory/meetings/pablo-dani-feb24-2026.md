## Key Decisions

### 1. Org Structure of Revenue
- **ADQ**: Volume + quality ratios only. NO revenue metrics.
- **Conversión**: Binary (did they monetize?). NO revenue.
- **LC / Ciclo de Vida**: ALL money, ALL revenue from FM onward.
- Reason: "Si yo veo un número de €15K en adquisición, ¿con qué lo comparo? No me sirve."

### 2. First Monetization (FM) Redefined
- FM is NOT a time threshold. It is an EVENT (can happen hours or months after registration).
- Do NOT measure median days to FM (confounded by Bitcoin bull/bear cycles).
- ADQ should measure: FM Rate (% who monetized) and Time-to-FM ratio (how fast).
- "Esto nos va a llevar a tener datos que no nos van a ayudar." — Pablo on 5-day median.

### 3. Lifecycle Framework = Activación + Retención
Pablo's iPad drawing (Feb 24, 16:32) shows:
```
Life Cycle → +TODOS
  Activación (circled) | Fase 1 | F2 | F3    → Wallet / Broker
  FM → (event)                                 → Loan / Earn
  X = 1 mes (activation window duration)
  acciones activación
  Fases Iterativas → D → +Éxitos
  Fase Despierta Hot → D+35días si... → boxes
```

**Activación phase:**
- Starts: at FM event
- Duration: X = ~1 month (not fixed, TBD with data)
- Actions: explain Loan, Pro, Earn, Card. Make incentive offers. Cross-sell.
- NOT Conversión's responsibility (Pablo García is technical, not lifecycle)

**Fases Iterativas (iterative phases):**
- Triggered by market events: "Mercado cayó 15% → offer Loan. Mercado subió → harvest"
- Also triggered by inactivity: Fase Despierta (35d), Fase Despierta Hot (60d)

### 4. "Fase Despierta" Naming
- Fase Despierta: user 35+ days without any action
- Fase Despierta Hot: user 60+ days without any action
- These are iterative phases, not fixed stages. Activate whenever criteria met.

### 5. Product-Specific Lifecycles

- Loan users: collateral-based trigger lifecycle
- Earn users: DCA path (60-70% lower churn — market stat, not Bit2Me internal)
