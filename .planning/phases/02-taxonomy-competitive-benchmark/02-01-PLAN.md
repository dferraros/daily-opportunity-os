**6.1 Taxonomy Overview**
- Purpose statement: this section defines the 6 trigger families that Phase 3 will score and tabulate

**6.2 Family A: User Configured**
- Definition: alerts the user explicitly creates (price above/below, % move, target reached, watchlist threshold)
- consent_category: CAT-USR, priority: P1, compliance_class: TRANSACTIONAL
- Entry criteria: user creates alert via Price_Alert_Set event
- Exit criteria: alert fires and is consumed (one-off) or remains active (persistent)
- Eligibility rules table: who_receives (any user who created the alert), who_never_receives (C8_Whale_Suppression, Excluded_Users), asset_scope (all listed assets, Layer 1)
- 4 example triggers using the template: A-01 Price Target Alert, A-02 Percentage Change Alert, A-03 Watchlist Price Move, A-04 LTV Threshold Alert (for Loan users)
- Channel: push primary, email fallback. Cooldown: 1 per asset per 4h
- Data source: Cloud Function + CoinGecko API (public data)
- Quiet hours: NOT exempt (unless user opts into 24/7 in preferences)

**6.3 Family B: Market Triggered**
- Definition: proactive alerts driven by market conditions, NOT user-configured
- consent_category: CAT-MKT, priority: P3, compliance_class: INFORMATIONAL
- Entry criteria: market condition detected (volatility > threshold, volume spike > 2x 30d avg, asset trending on social/exchange, price breakout)
- Exit criteria: market condition normalizes (below threshold for 4h)
- Eligibility: who_receives (users with consent_marketing_push=true AND who hold or have on watchlist the affected asset), who_never_receives (C8, Excluded, fatigue_risk > 0.7, users who dismissed B-family in last 7d), asset_scope (Top 50-100 by Bit2Me volume -- NOT all 420+ assets)
- 4 example triggers: B-01 Volatility Spike, B-02 Volume Spike, B-03 Trending Asset, B-04 Price Breakout
- Channel: push primary, in-app secondary. Cooldown: 1/day, 3/week
- CRITICAL: MiCA Art. 87-92 market abuse constraints apply. Data source MUST be public (CoinGecko API). All eligible users receive simultaneously (no staggering). Audit log required.
- Copy constraints: state facts only ("BTC is up 10% in 24h"), never suggest action

**6.4 Family C: Behavioral**
- Definition: triggers based on in-app user behavior patterns
- consent_category: CAT-MKT or CAT-PRO depending on trigger, priority: P3-P4, compliance_class: MARKETING
- Entry criteria: behavioral pattern detected (viewed asset 3+ times without purchase, deposited without trading within 72h, started order but abandoned, repeated login without action)
- Exit criteria: user completes the suggested action OR 30 days pass
- Eligibility: who_receives (users matching behavioral pattern with marketing consent), who_never_receives (C8, Excluded, users in active journey, fatigue_risk > 0.7)
- 4 example triggers: C-01 Watched Not Bought, C-02 Deposit No Trade (72h), C-03 Abandoned Order, C-04 Repeated Views No Action
- Data source: CleverTap Live Segments (real-time behavioral targeting) or BigQuery session analysis
- Channel: in-app primary (lower permission risk), push secondary. Cooldown: 1/week per pattern
- Copy constraints: describe behavior factually, do not imply opportunity
