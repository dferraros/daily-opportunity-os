**Structure:**

Section header: `## 10. Scoring System Architecture`
Purpose block, owners (Alvaro for BigQuery, Katy for CleverTap targeting, Daniel for weight calibration).

**10.1 Scoring Pipeline Overview**
- Diagram: BigQuery scheduled query -> user_scores table -> Hightouch 30min sync -> CleverTap user profile -> Campaign targeting
- All scores output 0-1 normalized using `LEAST(1.0, GREATEST(0.0, ...))` pattern
- Reference Phase 1 fatigue formula (Section 2.5) as the proven precedent for this pipeline

**10.2 SCORE-01: Market Relevance Score**
- Inputs: pct_change_24h, avg_pct_change_30d, stddev_pct_change_30d, volume_24h, avg_volume_30d
- Components:
  - price_zscore_normalized = ABS(pct_change_24h - avg_pct_change_30d) / NULLIF(stddev_pct_change_30d, 0) / 4.0 -- 2-sigma move = 0.5, 4-sigma = 1.0
  - volume_anomaly_normalized = (volume_24h / NULLIF(avg_volume_30d, 0) - 1.0) / 3.0 -- 2x avg = 0.33, 4x avg = 1.0
- Weights: price_zscore 0.60, volume_anomaly 0.40
- Combined formula as BigQuery SQL CREATE OR REPLACE VIEW (use exact SQL from 03-RESEARCH.md lines 244-266)
- Asset scope: WHERE tier IN ('T1', 'T3') AND median_daily_volume_eur_30d > 10000
- Evaluation frequency: Hourly
- Threshold calibration table by asset tier:
  - T1 (BTC/ETH): default 5% in 24h, z-score trigger 2.0 sigma
  - T2 (Stablecoins): 2% de-peg (absolute, not z-score)
  - T3 (Mid-cap): default 8%, z-score 2.0 sigma
  - T4 (Memecoin): user-configured only, no proactive

**10.3 SCORE-02: User Asset Affinity Score**
- Inputs: balance_eur, on_watchlist, days_since_last_trade, views_last_30d
- Components:
  - holdings_score = CASE WHEN balance_eur > 0 THEN LEAST(1.0, balance_eur / 500.0) ELSE 0.0 END -- EUR 500+ = max
  - watchlist_score = CASE WHEN on_watchlist THEN 0.5 ELSE 0.0 END
  - trade_recency_score = LEAST(1.0, GREATEST(0.0, 1.0 - (days_since_last_trade / 90.0))) -- 0 days = 1.0, 90+ days = 0.0
  - view_frequency_score = LEAST(1.0, views_last_30d / 10.0) -- 10+ views = max
- Weights: holdings 0.40, watchlist 0.20, trade_recency 0.25, views 0.15
- BigQuery SQL CREATE OR REPLACE VIEW (use exact SQL from 03-RESEARCH.md lines 279-308)
- This is per-user per-asset (not per-user global)

**10.4 SCORE-03: Trigger Opportunity Score**
- Formula: trigger_opportunity = market_relevance_score * user_asset_affinity_score * trigger_family_weight
- MULTIPLICATIVE, not additive -- any zero component kills the trigger
- trigger_family_weight table:
  - Family A: 1.0 (user requested, always fires)
  - Family B: 0.8 (market data, moderately selective)
  - Family C: 0.6 (behavioral, more selective)
  - Family D: 0.9 (lifecycle, high value when it fires)
  - Family E: 0.5 (cross-sell, most selective to protect permissions)
  - Family F: 1.0 (protective, always fires)
- BigQuery pseudocode joining score_market_relevance and score_user_asset_affinity

**10.5 SCORE-04: Notification Pressure Score**
- Extends Phase 1 fatigue formula (Section 2.5) with per-family tracking
- Inputs: notifications_sent_7d, notifications_opened_7d, notifications_dismissed_7d, per_family_sent_7d
- Formula:
  - global_pressure = notifications_sent_7d / 5.0 (normalized against 5/week safe ceiling)
  - family_pressure = per_family_sent_7d / family_weekly_cap (family-specific normalization)
  - open_rate_inverse = 1.0 - (notifications_opened_7d / GREATEST(notifications_sent_7d, 1))
  - notification_pressure = global_pressure * 0.40 + family_pressure * 0.30 + open_rate_inverse * 0.30
- Higher pressure = less likely to send. Used as (1 - notification_pressure) in Send Score Final.
- BigQuery pseudocode

**10.6 SCORE-05: Fatigue Risk Score**
- EXACT formula from Phase 1 Section 2.5 (DO NOT CHANGE -- already in production design):
  ```
  fatigue_risk = (notifications_sent_7d / 5) * 0.4
               + (notifications_dismissed_7d / max(notifications_sent_7d, 1)) * 0.3
               + (days_since_last_open / 7) * 0.3
  ```
- Thresholds: GREEN < 0.3, AMBER 0.3-0.7, RED 0.7-0.9, CRITICAL > 0.9
- Tier suppression rules (from Phase 1 Section 2.5):
  - GREEN: all tiers send
  - AMBER: suppress P5
  - RED: suppress P3+ (48h)
  - CRITICAL: suppress P2+ (7 days)
- Cross-reference: This score feeds Gate 2 of Send Score Final
- Note: "This formula is unchanged from Phase 1 Section 2.5. It is reproduced here for completeness and cross-reference."

**10.7 SCORE-06: Cross-sell Eligibility Score**
- Inputs: product_gap_count (how many products user does NOT use from eligible set), balance_tier (1-4), lifecycle_weight
- Components:
  - product_gap_normalized = product_gap_count / 5.0 -- max 5 cross-sell-eligible products (Earn, Pro, Card, Loan, Space Center)
  - balance_relevance = LEAST(1.0, total_balance_eur / 1000.0) -- EUR 1000+ = max
  - lifecycle_weight = CASE lifecycle_stage WHEN 'ACTIVE' THEN 1.0 WHEN 'POWER' THEN 0.8 WHEN 'FM' THEN 0.9 WHEN 'AT_RISK' THEN 0.3 ELSE 0.1 END
- Weights: product_gap 0.40, balance_relevance 0.35, lifecycle_weight 0.25
- BigQuery pseudocode
- V1 constraint: Cross-sell copy must use product awareness framing only (Phase 2 decision). No return comparisons.

**10.8 SCORE-07: Churn Risk Score**
- Inputs: days_since_last_action, actions_last_30d, actions_prev_30d, balance_eur_current, balance_eur_30d_ago
- Components (higher = higher churn risk):
  - recency_risk = days_since_last_action / 90.0 -- 90+ days = max risk
  - frequency_decline_risk = 1.0 - (actions_last_30d / NULLIF(actions_prev_30d, 0))
  - balance_decline_risk = (balance_eur_30d_ago - balance_eur_current) / NULLIF(balance_eur_30d_ago, 0)
- Weights: recency 0.40, frequency_decline 0.30, balance_decline 0.30
- BigQuery SQL CREATE OR REPLACE VIEW (use exact SQL from 03-RESEARCH.md lines 316-350)
- Churn risk is used as a BOOST in Send Score Final (high churn risk = MORE reason to send lifecycle triggers)

**10.9 SCORE-08: Send Score Final (Gated Architecture)**

THIS IS THE MOST CRITICAL SECTION. Structure it as:

**Gate Architecture Diagram:**
```
GATE 1: Compliance (binary) -----> BLOCK if fails
GATE 2: Fatigue (threshold) -----> BLOCK if fails
GATE 3: Cooldown (temporal) -----> BLOCK if fails
  |
  v (all gates pass)
WEIGHTED SCORE (0-1) -----> Compare to family threshold
  |
  v (above threshold)
SEND
```

**Gate 1: Compliance Gate (binary pass/fail)**
- Is user in C8_Whale_Suppression or Excluded_Users? -> BLOCK
- Does trigger require consent user hasn't given? -> BLOCK (check consent_marketing_[channel] for P2-P5)
- Is trigger classified ADVISORY_RISK? -> BLOCK (V1 -- all ADVISORY_RISK deferred)
- Is user in active CleverTap journey AND trigger is Family C/D/E? -> BLOCK (active_journey IS NULL check)
- ANY gate 1 failure = score is 0, notification does NOT send

**Gate 2: Fatigue Gate (threshold)**
- Read fatigue_risk score (SCORE-05)
- Compare against tier threshold:
  - P3 campaigns: fatigue_risk < 0.7
  - P4 campaigns: fatigue_risk < 0.7
  - P5 campaigns: fatigue_risk < 0.3
- P0-P2 are exempt from fatigue gate

**Gate 3: Cooldown Gate (temporal)**
- Has this trigger family sent to this user within the cooldown window?
- Check last_sent_[family] timestamp
- Cooldown values per family (from Phase 2 trigger definitions)

**Weighted Score (only computed if ALL gates pass):**
```
send_score = trigger_opportunity_score * 0.35
           + user_asset_affinity * 0.25
           + (1 - notification_pressure) * 0.20
           + (1 - fatigue_risk) * 0.10
           + churn_risk_boost * 0.10
```

**Family-specific overrides:**
- Family A: send_score = 1.0 (always sends, gated by cooldown only)
- Family B: standard formula, but bypass active_journey check in Gate 1
- Family E: replace trigger_opportunity with cross_sell_eligibility
- Family D: replace user_asset_affinity with lifecycle_urgency (CASE on lifecycle stage)
- Family F: send_score = 1.0 (always sends, protective)

**Minimum send score thresholds by family:**
| Family | Minimum Send Score | Rationale |
|--------|-------------------|-----------|
| A | N/A (always sends) | User requested |
| B | 0.40 | Time-sensitive market data |
| C | 0.50 | Behavioral nudges less urgent |
| D | 0.35 | Lifecycle transitions rare and high-value |
| E | 0.55 | Highest bar to protect push permission |
| F | N/A (always sends) | Protective |

BigQuery pseudocode for the complete Send Score Final computation as a single query joining all component scores.
