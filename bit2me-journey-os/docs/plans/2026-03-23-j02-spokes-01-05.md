## Reglas de convivencia Hub + Spokes {#reglas-de-convivencia}

Estas reglas gobiernan como interactuan el Hub J02-CORE y todos los Spokes. Katy debe configurarlas como global frequency rules en CleverTap antes de activar cualquier spoke.

### Tabla de reglas

| Regla | Valor | Razon | Owner |
|-------|-------|-------|-------|
| Max emails por semana por usuario | 2 | Klaviyo benchmark: >2/semana = +35% unsubscribe rate | Katy |
| Max push por semana por usuario | 2 | CleverTap fintech best practice | Katy |
| Gap minimo entre mensajes mismo canal | 72h | Evitar fatiga de canal | Katy |
| Hub pausa cuando Spoke activo | Si — inmediato | No doble mensajeria, no mensajes duplicados | Katy |
| Hub reanuda cuando Spoke termina | Si — automatico | Usuario vuelve a flujo principal si no convirtio | Katy |
| Exit de Spoke si usuario convierte | Inmediato (evento de conversion detectado) | No empujar producto que ya tiene | Katy + Alvaro |
| Solo 1 Spoke activo por usuario a la vez | Si | Prioridad descrita abajo | Katy |
| Holdout global | 10% de MMU (aprox 2.300 usuarios) | Grupo control para medir impacto total del sistema | Alvaro + Daniel |

### Prioridad de Spokes (cuando varios son elegibles simultaneamente)

```
SP-01 Earn > SP-02 Pro > SP-03 DCA > SP-04 Diversify > SP-05 Referidos
```

Logica: mayor impacto monetario primero. Earn tiene 72.4k usuarios elegibles y EUR 19.5M AUC dormant. Referidos solo aplica post-segunda compra por definicion.

### Como funciona la pausa del Hub

Cuando un Spoke se activa:
1. CleverTap marca al usuario como "spoke_active = true"
2. El Hub detecta este flag y pausa el envio del siguiente touchpoint
3. El Spoke corre su secuencia completa (E1, E2, E3 segun spoke)
4. Al terminar el Spoke (conversion O expiracion), se limpia el flag
5. El Hub reanuda desde el touchpoint donde lo dejo (no desde el principio)

**Condicion de expiracion:** cada spoke tiene su duracion maxima documentada abajo. Si el usuario no convierte en ese tiempo, el spoke expira, se resetea el flag y el Hub continua.

## SP-01 — Earn Activation {#sp-01-earn}

### Resumen ejecutivo

- **Prioridad:** #1 del sistema. Esta es la prueba mas importante.
- **Segmento elegible:** hasta 72.4k usuarios con EUR 19.5M AUC dormant
- **Producto:** Bit2Me Earn, 14-15% APY
- **Benchmark de referencia:** Coinbase staking post-email +23% activation con balance personalizado; Braze fintech cross-sell 6-12% activation
- **Principio psicologico:** Loss Aversion (Kahneman) — el dolor de perder es 2x mas intenso que el placer de ganar. Ver seccion A/B.

### Trigger de entrada

```
balance_idle_days >= 7
AND earn_product_active = false
AND user_status IN ('ACTIVE', 'POWER', 'AT_RISK', 'FM')
AND NOT IN holdout_global_10pct
```

**Fuente de datos:** BigQuery Gold Layer V0a (Alvaro). Campo `earn_product_active` requiere ser expuesto en el evento CleverTap — verificar con Alvaro antes de activar.

### Condicion de salida (exit)

- **Exit por conversion:** evento `earn_activation_confirmed` recibido — salida inmediata, no enviar emails restantes
- **Exit por expiracion:** 14 dias desde activacion del spoke sin conversion
- En ambos casos: flag `spoke_active` se limpia, Hub reanuda

### Interaccion con Hub

- Si Hub activo: Hub pausa al activarse SP-01. SP-01 corre su secuencia. Hub reanuda al terminar.
- Si Hub no activo (usuario ya paso por el Hub): SP-01 se activa igualmente si trigger se cumple.

### A/B Test principal — SP-01

**Este es el test mas importante del sistema J02.**

| Campo | Detalle |
|-------|---------|
| Nombre del test | SP01-EARN-FRAMING |
| Hipotesis | La Variante A (loss framing) genera mayor tasa de activacion que la Variante B (gain framing) por efecto de Loss Aversion (Kahneman, Prospect Theory). El dolor de perder valor es psicologicamente 2x mas intenso que el placer de ganar el mismo valor. |
| Variante A | Loss framing: "Tus [ASSET] llevan [N] dias parados. Podrian haber generado EUR [Y]." |
| Variante B | Gain framing: "Activa Earn y genera [APY]% anual sobre tu [ASSET]." |
| Muestra minima | 2.000 usuarios por variante (4.000 total). Tenemos volumen suficiente con 72.4k elegibles. |
| Duracion | 14 dias |
| Metrica primaria | Earn activation rate a 14 dias post-email (E1) |
| Metrica secundaria | Open rate, click rate por variante |
| Nivel de confianza requerido | 95% (p < 0.05, Welch t-test) |
| Benchmark esperado | 6-12% activation rate (Braze fintech cross-sell benchmark) |
| KPI target | >8% activation rate |
| Hipotesis de resultado | A gana por 1.5x-2.0x. Si B gana, revisar segmento — posible que gain framing funcione mejor en usuarios nuevos (<60 dias) vs usuarios con balance alto. |
| Owner | Daniel (diseno) + Katy (setup CleverTap) |

**Nota para Katy:** este test va en E1 unicamente. E2 y E3 usan copy neutro (el que no tenga A/B) para no contaminar el test con multiples variantes activas en paralelo.

## SP-03 — DCA / Inversion Recurrente {#sp-03-dca}

### Resumen ejecutivo

- **Segmento:** usuarios que visitan pero no operan — tienen intencion pero les paraliza el timing
- **Trigger:** login_without_trade >= 3 en los ultimos 14 dias AND sin inversion recurrente activa

- **Benchmark:** Coinbase DCA users — 2.3x higher 90-day retention vs one-time buyers (PLG analysis, 2024).

### Trigger de entrada

```
login_without_trade_14d >= 3
AND recurring_buy_active = false
AND user_status IN ('FM', 'ACTIVE', 'AT_RISK')
AND spoke_active = false
AND NOT IN holdout_global_10pct
```

## SP-04 — Diversificacion {#sp-04-diversify}

### Resumen ejecutivo

- **Segmento:** usuarios con portfolio de un solo activo, mas de 30 dias desde la primera compra
- **Trigger:** single_asset_portfolio = true AND days_since_first_purchase >= 30
- **Producto:** segunda compra de activo diferente (cross-sell de catalogo)
- **Principio psicologico:** Risk Aversion + Availability Heuristic. "No pongas todos los huevos en una cesta" es universalmente comprendido. Hacer la diversificacion sentir como comportamiento normal (el 73% ya lo hace) reduce la barrera de accion.

| Altcoin (cualquier) | BTC | Ancla de seguridad — menor riesgo sistemico |

## SP-05 — Programa de Referidos {#sp-05-referidos}

### Resumen ejecutivo

- **Segmento:** usuarios que acaban de hacer su segunda compra, 48h despues del evento
- **Producto:** programa de referidos Bit2Me (codigo propio + incentivo doble)
- **REGLA CRITICA: NUNCA activar antes de la segunda compra confirmada.** Ver nota abajo.
- **Principio psicologico:** Peak-End Rule (Daniel Kahneman) — el momento post-segunda compra es un pico de satisfaccion. Es el momento optimo para pedir. Reciprocidad (Cialdini) — el usuario ha recibido valor dos veces. El acto de compartir se siente natural, no forzado.
- **Benchmark:** Binance referral program — 35% de nuevos usuarios vienen de referidos cuando se pregunta post-second-trade (Binance growth case study, 2023).

### Por que NUNCA antes de la segunda compra

La primera compra es un momento de incertidumbre. El usuario no sabe todavia si la experiencia vale la pena. Pedir un referido antes de que el usuario haya consolidado su inversion es prematuro y genera tension cognitiva ("me estan pidiendo que recomiende algo que yo mismo no he probado bien").

La segunda compra es diferente: es un acto de confianza. El usuario ha evaluado la plataforma, ha decidido volver. Ese momento de satisfaccion es el unico momento correcto para pedir que recomiende.

**En CleverTap: el trigger debe ser el evento `second_purchase_confirmed`, no el primero.**

### Trigger de entrada

```
second_purchase_confirmed = true
AND referral_spoke_triggered = false (nunca repetir)
AND 48h elapsed since second_purchase_confirmed
AND NOT IN holdout_global_10pct
```

**Nota importante:** este spoke se activa una sola vez por usuario. Si ya se activo anteriormente (aunque no haya compartido), no volver a activar. Campo `referral_spoke_triggered` debe setearse en BigQuery/CleverTap al activar el spoke.

### Condicion de salida

- **Exit por conversion:** evento `referral_link_shared` O `referral_link_clicked`
- **Exit por expiracion:** 5 dias
